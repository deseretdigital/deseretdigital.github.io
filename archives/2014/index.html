Metrics at DDM

At DDM we’re lucky to stand on the shoulders of giants in regards to collecting and monitoring the metrics we collect. If you’ve never read the post by Etsy’s team on Measure Anything, Measure Everything, go read that first then come back.

We’ve tried to adopt Etsy’s attitude on measuring all different aspects of our application and then use graphs to monitor said data. One of the challenges is that for a company our size, we do a significant amount of traffic, so the costs of a tool like New Relic would be significant. We also have a pretty locked down internal environment where our servers live, so it is easier for us to implement something internally.

One important thing for us is low maintenance and ease of use. We don’t have a dedicated “DevOps” team, and our Dev Team size is small compared to the # of products & traffic we serve. So what we’ve implemented should work for the smallest of developer shops & websites.

Our Setup

We use the following tools to track and measure system and application metrics:

  • StatsD - Network Daemon that runs on Node.js that collects metrics and timings, aggregates them, and sends them to Graphite.
  • Graphite - Stores real-time data & outputs rendered graphs or json data.
  • Grafana - An open source Dashboard & Graph Editor. Uses ElasticSearch as it’s backend, very lightweight.

We send all of our metrics over UDP to StatsD. StatsD will then collect that data flush it to Graphite every second. We then use the Graphite interface to browse our data or quick one-off graphs. Once we know we have the data we want, we’ll create graphs in Grafana. Pretty darn simple.

Getting Data In

Once we had our StatsD / Graphite / Grafana combo setup, we needed to get data into it. Thankfully StatsD is popular enough that there are many clients for just about any language out there.

Application Metrics in PHP

In our PHP Apps we use the domnikl/statsd package, though there are many others out there.

We wrap our StatsD instance in a singleton (gasp! I know) that is autoloaded. The reason is you end up sprinkling StatsD stuff all over your application without worrying about getting it setup properly. You definitely could use some more proper method of dependency injection, but for this particular case it worked well.

Timings

Timings are crucial for capturing when releases or changes improve or worsen the performance of our website. A timing is super simple to do:

  • Get the start time in milliseconds
  • Do whatever code / action
  • Get the end time in milliseconds
  • Compare & send total milliseconds elapsed to StatsD

Capturing Micro Events

There are things that happen a lot on our application, but other times things that don’t happen very often. One example is we scenario where our contributor system will push over a story to our content system. That event only happens ~40 times a day, but it is a very important that the system works. The same thing could be for user registrations, or comments, or any other thing.

The problem is Graphite/StatsD work by default in a “per second” view. Then when the second data is rolled up into a 10 second or minute granularity, you get things like lots of 0’s and then a 0.01 and then some more 0s. Its because you have 0.01 of these events per second.

Graphite has an awesome function to combat that called “hitcount” which you can pass in a time and it won’t tell you how many per second, but how many of an event occurred. Here is a graph of the hitcount per 5 minutes:

This is much easier to see if I go a full hour without stories crossing, I know I need to take a look.

Slave Replication

Using our statsd-mysql-watcher we can have the master report it’s current binary log position, and the slaves report theirs. We’ve had problems in the past where the connection between our masters and slaves would die, but they won’t realize it. The master would update, but since the slave didn’t have a real connection it wasn’t getting those. So if you did a seconds_behind_master, it would report 0.

Here we are showing the current binary log position of the master and of the two slaves. You can see they are uniform, which means they are all on the same position, or very close. However you can see Slave 1 deviates and doesn’t follow the other two. It’s replication had a hiccup and was stuck because the connection died. Once we realize that we were able to fix the connection. You can see it moved to get back in line once fixed, and once it was following the others we knew it was completely caught up.

Automating staging of feature branches

Have you found team trying to swtich your staging server from one branch to another for a demo of feature X or Y? Why limit your project to just one staging site? Just make all your branches available on your stage server as soon as they are pushed to github.

We have a simple to use script that can quickly pull all of your branches down to your server. Throw in a little server and DNS configuration and each new branch will be accessable as its own staging site.

Lets say you work on example.com and have the following branches - social, login and master. Using 4gitphull you’ll be able to access social.example.com, login.example.com and master.example.com.

4gitphull can also generate a static html file that contains a list of commits that are on a branch, but not in master. Providing that your site has a page that will show the hash of the current deployed code, a second file can be generated that lists the commits that are on master, but are not yet live. Who wouldn’t want to know what would go live if you were to deploy right now?
